Linux下安装git

前言

早几年时候买的阿里云或腾讯云的服务器，默认的 centos 的版本自带的有 git，现买的服务器系统中没有自带 git，又想使用高版本 git 只能手动安装了！现记录一下安装步骤！

内容

1、安装包下载 wget -c https://github.com/git/git/archive/v2.29.0.tar.gz
2、压缩包解压 tar -zxvf v2.29.0.tar.gz
3、安装编译源码所需依赖 yum install curl-devel expat-devel gettext-devel openssl-devel zlib-devel gcc perl-ExtUtils-MakeMaker 但：我执行之后出现提示如下：

这说明系统有这些依赖。

4、安装依赖时，yum 自动安装了 Git，需要卸载旧版本 Git，命令为： yum remove git 出现提示输入 y 即可！注：我的 linux 上执行了上面的第三步骤，并没有 git

5、进入解压后的文件夹，命令 cd git-2.29.0 ，然后执行编译，命令为 make prefix=/usr/local/git all 耐心等待编译即可；

6、安装 Git 至/usr/local/git 路径，命令为 make prefix=/usr/local/git install

7、打开环境变量配置文件，命令 vim /etc/profile ，在底部加上 Git 相关配置信息：PATH=$PATH:/usr/local/git/bin export PATH 然后保存，退出！

8、输入命令 git –version ，查看安装的 git 版本，校验通过，安装成功，截图如下：

问题

1、在 linux 上依次设置：git config –global user.name “zhangbo” git config –global user.email “1174839512@qq.com“

2、ssh-keygen -t rsa -C “1174839512@qq.com“ 提示：-bash: ssh-keygen -t rsa -C 1174839512@qq.com: command not found

3、一顿搜索啊，找到了解决 bash: ssh-keygen: command not found

4、按照上面帖子执行：yum install openssh，可是还是不行啊，Google 了很多，后来我发现有一个指令：ssh-keygen -t rsa -b 2048 -C “1174839512@qq.com“ 与我的有道云笔记报错（ssh-keygen -t rsa -C “1174839512@qq.com“ ）的内容不一样！后来就死马当成活马医！结果可以了！

5、接着说解决上面问题的过程：既然是一直提示-bash: ssh-keygen -t rsa -C 1174839512@qq.com: command not found，那我就键入 ssh 看提示啥，效果如下：

细看一下，没有 -C 的选项，但是第二个指令也有 -C，那我就先试试这个：ssh-keygen -t rsa -b 2048 -C “1174839512@qq.com“，竟然可以了！猜想：linux 下必须要设置 -b 吧！Windows 估计是读取的默认的配置！问题解决！

Nodejs如何在linux上安装

前言

虽然一直在使用 nodejs 启动 spa 项目，但是在服务器上部署 nodejs 作为后端，做的很少，记录一下步骤！

内容

1、本地机器、公司的 Jenkins 发版的机器记得保持版本号一直！
2、直接打开官网，可切换中文查看文档（就没有必要使用nodejs 中文网地址了）。在站点找到：下载 => 以往的版本 => version list（找自己的 Linux 对应的版本）=> 我本地机器是（Win10 v12.18.3）

Linux 对应的版本截图如下：

点击下载，详情如下：

找到具体版本对应的平台包，如 v12.18.3 对应的 Linux 平台的包：node-v12.18.3-linux-x64.tar.xz，注意针对上面的红色框圈出来的文件，不要点击，要单击鼠标右键，找到“复制链接地址”，如下图所示：

我在自己的服务器上切换到目录为：~

这时候执行： wget -c https://nodejs.org/download/release/v12.18.3/node-v12.18.3-linux-x64.tar.xz，随后提取文件：tar -xvf node-v12.18.3-linux-x64.tar.xz，修改名字：mv node-v12.18.3-linux-x64 ~/nodejs12.18.3。然而，这种方式安装的 Nodejs 并不完美。首先，Nodejs 的命令 node 和 npm 并不能在全局使用。为了解决这个问题，我们需要建立两个软链接：ln -s ~/nodejs12.18.3/bin/node /usr/local/bin/node、ln -s ~/nodejs12.18.3/bin/npm /usr/local/bin/npm。这时候我们执行：npm -v、node -v，就可以显示 npm node 的版本信息了！

趁热打铁就随便安装一下 yarn 吧，我本机、公司安装的都是 1.22.4。于是我执行：npm i yarn@1.22.4 -g。安装效果如下：

此时的 yarn 不能用，因为没有配置环境变量！这时候我想起来了，我的 Windows 上 yarn 安装在C:\Users\zhangboba\AppData\Local\Yarn\bin下，我把该目录添加到了我的环境变量！那我的 Linux 安装的 yarn 是不是需要添加环境变量啊，查资料之后，得知添加环境变量方法：找到 vim /etc/profile，添加：export NODE_HOME=~/nodejs12.18.3/bin、export PATH=$NODE_HOME:$PATH，保存退出即可！至此 node 安装完毕！

2、componentWillMount：最初该钩子是为了异步的获取数据，现在官方已不赞成使用（在 ssr 场景下，服务端与客户端都会渲染一遍，浪费资源！）

3、componentDidMount：该组件是渲染完毕之后会触发！此时页面中已经有相关的 DOM，我们可以进行 DOM 的相关操作（思考：我们使用了 MVVM 之后还会操作 DOM 吗？）

我们使用这个生命周期一般用于优化，有的时候，我们在界面上只需要更新一个很小的组件，而一个父组件的更新会造成整个子组件都进行渲染，形成一个崭新的虚拟 DOM，但是这样的话，会造成资源的浪费，我们可以根据实际的开发情况在 shouldComponentUpdate()生命周期加入条件，来进行性能的优化。

前言

yarn 安装有三种方式（官网说），最便捷的还是使用 npm i yarn@1.22.4 -g，默认安装目录：C:\Users\admin\AppData\Roaming\npm\yarn（Win 下执行 where yarn 可知）。yarn 诞生就是为了解决 npm 早期依赖包比较混乱的问题，伴随着 node 的升级，npm 版本也随之升级，针对依赖包已经做的很好了，详情可点击我。

指令

1、根据上面疑问版块的内容，可知，yarn 指令有：

  • yarn add pkgName
  • yarn add pkgName@1.22.4
  • yarn remove pkgName
  • yarn global add pkgName
  • yarn global remove pkgName
  • yarn config ls
  • yarn global ls

2、通用指令：

  • yarn config [options]
  • yarn global [options]

其实很多的指令其实就是通用指令加上子命令而来！假如我们记不清楚指令了，我们可键入 yarn global 会提示你缺少子命令，同时把相关常用的子命令给你罗列出来！截图如下：

3、yarn upgrade 仅更新 yarn.lock、package.json 不更新，参考：升级包到它们基于规范范围的最新版本、yarn upgrade 更新依赖包时 yarn.lock 更新但 package.json 不同步更新版本信息

9、dependencies 和  devDependencies 相同的地方： 就是在你或者别人  clone  这个库进行开发调试的时候，可以通过  npm install  一键安装这两个目录下所有的依赖，而不用去一行行找你到底在文件中引用了那些依赖。不同的地方在于： 当某个项目依赖了你的   mylib，那么在安装的时候会链式地安装  mylib  这个项目中  dependencies  配置声明的依赖，因为根据约定，npm  认为这是项目运行时需要的依赖。而  devDependencies  则是开发时需要的依赖。  如果你输入  npm install –production  指令，就只会安装  dependencies  目录下的依赖，在一些服务端自动构建的过程中或者在一些特殊的需求下，可能会用到这个指令。以上总结缘由是，场景一：针对 node-ruanmou 项目中，在其 package.json 中仅有 "dependencies": {"express": "^4.17.1"} 于是 npm i，在 node_modules 下有许多的依赖包，我于是找到了 D:\Application\node\node-ruanmou\node_modules\express\package.json，发现其 dependencies 中仅有依赖数量 30 个，可是我的 D:\Application\node\node-ruanmou\node_modules 下面的依赖包除了 .bin 目录 express 还有 48 个包，这说明 express\package.json 下的 dependencies 中的依赖包，也依赖了其他包，在执行 npm i 时候一并下载了！场景二：我在 vue-cli-raw 中故意安装 ElementUI 依赖到 devDenpendencies，页面中仅使用了 el-button，构建之后，在 dist 目录下起服务，按钮效果完好展示。带着这种疑问找到了上面的解答！
