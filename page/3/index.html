代码片段之Eslint

解析器

解析器有很多，但兼容eslint的解析器有以下几个：

  • Espree：默认解析器，一个从Esprima中分离出来的解析器，做了一些优化

  • Esprima：js标准解析器

  • Babel-ESLint： 一个对Babel解析器的包装，babel本身也是js解析器的一种（不然怎么能转化为兼容性代码呢？首先需要进行js解析，才能转化）。如果我们的代码需要经过babel转化，则对应使用这个解析器

  • typescript-eslint-parser(实验) - 一个把 TypeScript 转换为 ESTree 兼容格式的解析器

链接

extends 与 plugins

  • 个人理解：extends 里面的配置，就是一些人写好的插件，我们不用写了，就直接告诉 eslint 我继承这些规则即可，而 plugins 的场景就是：有些规则不满足你的需求，你想私人定制，于是自己写了插件，在此处配置后 eslint 就会解析你的插件！

  • 可共享的配置可以省略包名的前缀 eslint-config-，即实际设置安装的包名是 eslint-config-pkgName

  • 从插件中获取的规则，书写规则为 「plugin:插件包名/配置名」如：plugin:vue/essential 是简写，其在 node_modules 里的包名是 eslint-plugin-vue

AJAX与JSONP

前言

记录以免忘记

AJAX

GET

1、针对 get 请求，我们可在浏览器地址栏键入请求地址，此时浏览器进行了二次封装！为啥这样说，在浏览器控制台 Elements 有元素节点，在 Network 中请求记录中，All 有可现实，使用 xhr 过滤则无记录！同样请求连接键入 Postman，选择 get 请求，则是正常请求！注：测试效果时，禁止 json 格式化等插件，以免造成干扰！在测试请求页的 title 有 favicon.ico。测试 get 请求链接：http://api.map.baidu.com/geocoder?location=38.76623,116.43213&output=json&key=r49TjsbOLT9nkbS4ZljYf1GgnTkecbNQ。

POST

1、POST 类型的请求，后台有时候会告诉前端：“不要用 body 方式传递参数，用 param 方式”。为何？猜，是后端框架解析参数方便所致！

JSONP

1、JSONP 需要后端配合，jq 对其进行了优化，只要是 jsonp 形式，无论 type 写成 post 与否皆转为 get。注：别忘记添加 dataType: 'JSONP';
